# SNL 编译器设计思路

本文档只记录当前编译流程中会用到的文件。Predict 集生成、LL(1) 表生成等辅助脚本不在主编译流程内，因此不写入本文档。

## 一、命名规则

当前主流程文件使用 `序号_功能名.py` 的形式命名：

- `1_`：词法分析阶段
- `2_`：递归下降语法分析阶段
- `3_`：语义分析阶段
- `4_`：测试入口

说明：`1_lexer.py` 这种文件名比 `1.lexer.py` 更适合按文件名定位和用 `importlib.import_module()` 加载。由于 Python 的普通 `import xxx` 语法不支持数字开头的模块名，所以代码中仍使用 `importlib` 加载这些带序号的模块。

## 二、编译流程文件

| 文件 | 作用 |
| --- | --- |
| `1_lexer.py` | 词法分析器。把 SNL 源代码转换成 Token 序列。 |
| `2_recursive_descent_parser.py` | 递归下降语法分析器。把 Token 序列转换成 AST 语法树。 |
| `ll1_parser.py` | LL(1) 表驱动语法分析器。作为语法分析的另一个可选方案，也把 Token 序列转换成 AST。 |
| `ll1_table.json` | LL(1) 分析表数据。`ll1_parser.py` 运行时读取该文件。 |
| `3_semantic_analyzer.py` | 语义分析器。检查作用域、重复定义、类型匹配、过程调用参数等语义规则。 |
| `snl_frontend_app.py` | 图形界面入口。自动调用词法分析、语法分析和语义分析，并展示结果。 |
| `4_semantic_tests.py` | 语义测试入口。用于验证分析流程是否正确。 |

## 三、主编译流程

```text
SNL 源代码
  |
  v
1_lexer.py
  Tokenizer.tokenize()
  生成 Token 序列
  |
  v
语法分析二选一
  |
  |-- 2_recursive_descent_parser.py
  |     syntax_analysis(tokens)
  |
  |-- ll1_parser.py
        Program(tokens)
        读取 ll1_table.json
  |
  v
生成 AST 语法树
  |
  v
3_semantic_analyzer.py
  semantic_analysis(program)
  |
  v
输出符号表或语义错误列表
```

## 四、GUI 执行流程

```text
snl_frontend_app.py
  |
  v
用户输入 SNL 源代码
  |
  v
1 秒无修改后自动分析
  |
  v
1_lexer.py 生成 Token 表
  |
  v
根据界面选项调用语法分析器：
  - 递归下降法：2_recursive_descent_parser.py
  - LL(1) 法：ll1_parser.py
  |
  v
3_semantic_analyzer.py 执行语义分析
  |
  v
界面显示 Token 表、语法树、语义错误或符号表
```

运行 GUI：

```bash
python .\snl_frontend_app.py
```

## 五、测试流程

```text
4_semantic_tests.py
  |
  v
1_lexer.py
  |
  v
2_recursive_descent_parser.py
  |
  v
3_semantic_analyzer.py
  |
  v
检查 expected_errors 是否出现
```

运行测试：

```bash
python .\4_semantic_tests.py
```

## 六、当前语义分析能力

`3_semantic_analyzer.py` 当前支持：

- 同一作用域内类型、变量、过程重复定义检查
- 过程参数重复定义检查
- 未声明变量检查
- 未定义类型检查
- 未定义过程检查
- 数组下标类型检查
- 记录字段存在性检查
- 赋值语句左右类型匹配检查
- 算术表达式操作数类型检查
- 关系表达式左右类型兼容检查
- 过程调用参数个数和类型检查
- `var` 参数必须传入变量的检查
